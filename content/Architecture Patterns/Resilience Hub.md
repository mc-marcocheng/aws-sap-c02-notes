# AWS Resilience Hub

AWS Resilience Hub gives you a central place to define, validate, and track the resilience of your AWS applications. It helps you understand if your application meets your Recovery Time Objective (RTO) and Recovery Point Objective (RPO) targets.

## Key Features

- **Resilience Assessment:** Analyzes deployed infrastructure (via CloudFormation, Terraform, Resource Groups, or EKS) against predefined resilience policies.
- **RTO and RPO Targets:** Define strict policies for RTO (how fast to recover) and RPO (how much data loss is acceptable) across application tiers, infrastructure, and availability zones.
- **Actionable Recommendations:** Provides code, configuration, and architectural recommendations to improve resilience.
- **SOPs and Alarms:** Automatically generates standard operating procedures (SOPs) as Systems Manager Automation runbooks, and creates CloudWatch alarms.
- **Continuous Validation:** Integrates with CI/CD pipelines to assess resilience before production deployment.

> [!info] Defining an Application
> In Resilience Hub, an application is a collection of resources. You define it by pointing Resilience Hub to an AWS CloudFormation stack, AWS Terraform state files, AppRegistry, or AWS Resource Groups.

## Integrations

- **AWS Fault Injection Service (FIS):** Resilience Hub tightly integrates with FIS to simulate real-world failures (e.g., AZ drop, database failover) and validate that the RTO/RPO targets are actually met under duress.
- **AWS Systems Manager (SSM):** Uses SSM Automation documents to define and execute SOPs for recovery.
- **Amazon CloudWatch:** Deploys alarms to monitor the health of the application based on the resilience assessment.

## Strategic Scenarios (SAP-C02)

### Scenario 1: Validating RTO/RPO for Mission-Critical Apps
**Challenge:** A financial company needs to guarantee that its payment processing application meets an RTO of 15 minutes and an RPO of 5 minutes across multiple AZs. They need continuous validation.
**Solution:** Define the application in **AWS Resilience Hub**, assign a resilience policy with the required RTO/RPO, run an assessment, and implement the recommended CloudWatch alarms and SSM SOPs. Finally, use **AWS FIS** experiments generated by Resilience Hub to inject faults and prove the application recovers within the RTO.

### Scenario 2: CI/CD Pipeline Integration
**Challenge:** A development team frequently deploys infrastructure changes that sometimes degrade the application's resilience.
**Solution:** Integrate AWS Resilience Hub into the CI/CD pipeline using the AWS CLI or API. Before deploying to production, run a resilience assessment against the pre-production environment. Fail the pipeline if the resilience score drops below a certain threshold or if the RTO/RPO targets are no longer met.

> [!exam]
> Watch for keywords like "centralized RTO and RPO tracking", "generate recovery SOPs", or "validate resilience posture". **Resilience Hub** is the primary service for assessing and tracking, while **FIS** is the service for actively testing (chaos engineering).
